# Windows Terminal Experience Parity Spec

## 1. 文档目的

本文档定义 `qt-terminal-widget` 后续演进的唯一体验目标：

**在不强制绑定特定内部架构的前提下，使该项目在核心终端使用场景中的交互体验、稳定性和可预期性达到可与 Microsoft Windows Terminal 对标的水平。**

这里的“对标”不要求：

- UI 视觉逐像素一致
- 设置系统、标签页、分屏等应用级功能完全一致
- 底层渲染技术必须相同

这里的“对标”要求：

- 在用户执行命令、浏览历史、编辑输入、选择复制、粘贴、滚动、运行全屏 TUI、处理 Unicode/IME、调整窗口尺寸时，**表现出的行为语义和用户体感必须与成熟终端一致**
- 对 `Claude Code`、`PowerShell`、`cmd`、`bash`、`vim`、`less`、`fzf`、`lazygit` 等典型工具，必须达到“用户不会因为终端宿主异常而明显降低效率”的程度

本文档是**产品体验合同**，不是建议列表。

- 本仓库维护者
- 后续接手该项目的 Agent
- 负责实现或验收终端行为的开发者

## 3. 核心结论

当前项目已经具备“可嵌入 Shell、可解析 ANSI、可显示文本”的基础能力，但距离“体验对标 Windows Terminal”仍有明显差距。差距不在于是否使用 Qt、是否 GPU 渲染、是否复用现成控件，而在于：

1. **终端的外部行为状态机尚未完整**
2. **视口、光标、输入、选择、滚动、备用屏之间仍存在耦合**
3. **很多行为是“修补式”条件分支，而不是统一的终端语义模型**

因此，后续实现必须优先补齐“体验语义”，而不是继续仅修复单点事件。

---

## 4. 非目标

以下内容不属于本文档的强制范围：

- 多标签页管理
- 分屏管理
- 配置 UI
- 配色方案编辑器
- 背景图片/GIF
- 应用级命令面板
- 与 Windows Terminal 完全相同的设置文件格式

这些功能可以后续补，但**不应优先于终端核心体验语义**。

---

## 5. 体验总目标

### 5.1 用户体验基线

实现完成后，以下主观描述必须成立：

- 用户在终端中输入时，不会感到光标、视口、滚动条、选择区互相打架
- 用户滚到历史位置时，终端不会因为新的输入、回显或点击而突然把视口拉回底部
- 用户回到底部后，终端会自然恢复跟随输出
- 用户运行 `Claude Code`、`PSReadLine`、`fzf`、`vim`、`less` 等程序时，不会因为宿主终端错误识别模式而导致输入、滚轮、光标、历史显示异常
- 用户面对中文、emoji、全角字符、组合字符、IME 输入时，不会频繁看到错位、覆盖、光标跳动、候选位置异常
- 用户缩放窗口时，终端历史和当前屏幕内容不会表现出明显“这是个文本编辑器伪装成终端”的痕迹

### 5.2 交互判断标准

如果用户在出现异常后会自然地说出以下任何一句，说明仍未达标：

- “滚一下就乱了”
- “光标又自己跑了”
- “看历史的时候一输入就跳底”
- “这个程序明明是行编辑模式，却被当成全屏程序”
- “复制粘贴和正常终端不一样”
- “中文/emoji 显示不稳”

## 6. 实现策略选项

本文档不强制唯一架构，但只接受以下三类实现路线中的任意一种。

### 方案 A：在现有 `QPlainTextEdit` 基础上继续演进

优点：

- 改动路径最短
- 现有代码可复用较多
- 短期内可以继续快速验证行为

缺点：

- 很多终端语义会持续与编辑器语义冲突
- 光标、选区、IME、滚动、视口的隔离难度高
- 后期复杂度容易快速上升

适用条件：

- 仅用于短期补齐 P0/P1 体验缺口
- 不应作为长期终态的默认假设

### 方案 B：保留 Qt Widget 外壳，但引入独立终端缓冲区与自绘层

优点：

- 可以保留 Qt 集成便利性
- 能把“缓冲区/视口/光标/选择”从编辑器语义中解耦
- 更容易实现真正的终端行为状态机

缺点：

- 需要分阶段替换现有逻辑
- 中期工作量明显大于方案 A

适用条件：

- 希望既保留嵌入式控件路线，又真正做出成熟终端体验

### 方案 C：重写为专用终端控件

优点：

- 边界最清晰
- 最容易从根上统一终端行为模型

缺点：

- 成本最高
- 短期回归风险最大

适用条件：

- 当现有 `QPlainTextEdit` 路线已经明显阻碍 P0/P1 体验目标时

### 推荐

**推荐采用方案 B。**

原因：

- 它不要求立刻推翻整个项目
- 它允许先保留 Qt 容器集成能力
- 它是最有可能在工程成本和体验结果之间取得平衡的路线
- 它能为后续进一步走向方案 C 预留空间

## 7. 必须具备的终端体验语义

本节是本文档最重要的部分。后续实现是否达标，以本节为准。

### 7.1 缓冲区与视口必须分离

必须具备以下语义：

- 终端输出缓冲区是独立概念
- 当前屏幕视口是独立概念
- 光标位置是独立概念
- 选择区域是独立概念

禁止出现以下现象：

- 因为设置文本光标位置而隐式改变视口
- 因为滚动条值变化而隐式改变终端逻辑光标
- 因为选区变化而让提示符位置失效

验收标准：

- 用户向上滚动查看历史后，新的输出到来时，若用户未处于底部跟随模式，则视口保持原位
- 用户回到底部后，新的输出自动跟随

### 7.2 必须具备明确的交互状态机

至少要有以下显式状态：

- `FollowOutput`
- `BrowseHistory`
- `Selecting`
- `AlternateScreen`
- `ComposingIME`

必须为每个状态定义：

- 进入条件
- 退出条件
- 滚轮行为
- 鼠标点击行为
- 光标显示行为
- 输入行为
- 视口行为

禁止继续使用“多个布尔值临时拼出模式”的无统一定义方式。

### 7.3 主屏与备用屏切换必须以终端语义为准

必须遵守：

- 显式收到 VT 备用屏切换序列时，进入或退出备用屏
- 不得仅依赖“光标隐藏 + 子进程存在 + 某个 Win32 模式”之类启发式规则决定是否切备用屏

可以保留启发式逻辑，但必须满足：

- 启发式逻辑默认关闭，或只作为有明确白名单/特征保护的兼容策略
- 不能误伤 `Claude Code`、`PSReadLine`、普通 REPL、半屏交互式程序

验收标准：

- `vim`、`less`、`fzf` 这类明确全屏程序进入备用屏后行为正常
- `Claude Code` 这类非全屏但带行编辑/动态刷新程序不会被错误切入备用屏

### 7.4 光标必须是独立终端光标，不得依赖编辑器插入点语义

必须满足：

- 光标显示/隐藏由终端状态控制
- 光标渲染位置由终端缓冲区坐标决定
- 浏览历史时，光标可隐藏或冻结，但不能抢视口
- 选择文本时，光标不应破坏选区感知
- IME 输入时，候选定位必须围绕终端光标语义而不是编辑器默认插入点

建议：

- 光标使用独立 overlay 绘制
- 不再依赖 `QPlainTextEdit` 的真实插入光标作为最终视觉结果

### 7.5 输入语义必须完整

必须覆盖：

- 普通字符输入
- Control 组合键
- Alt/Meta 组合键
- Home/End
- PageUp/PageDown
- Insert/Delete
- Bracketed paste
- IME `preedit` 与 `commit`
- 多行粘贴

必须避免：

- 粘贴内容被当成逐键输入导致行为异常
- IME 候选框位置错误
- 组合键在不同 shell 中行为不一致到影响基本使用

重点验收程序：

- PowerShell + PSReadLine
- `cmd`
- `bash`
- `python`
- `node`
- `Claude Code`

### 7.6 滚动体验必须终端化

必须满足：

- 主缓冲区下，滚轮默认用于浏览历史
- 备用屏下，只有在程序显式开启鼠标协议时，滚轮才转发为终端鼠标事件
- 备用屏下若程序未声明接管滚轮，则不得随意伪造方向键吞掉用户意图
- 用户滚到历史区后，单击、输入、回显、右键等动作不得无条件把视口拉到底部

必须定义：

- “回到底部”的准确条件
- 何时恢复 `FollowOutput`
- 何时保持 `BrowseHistory`

### 7.7 选择、复制、粘贴必须符合成熟终端直觉

必须满足：

- 拖拽选区时，不扰动终端光标状态
- 复制后是否清除选区必须是明确设计，不得是偶然副作用
- 有选区时右键、无选区时右键、Shift+右键、Ctrl+Shift+C/V 等行为必须一致、可预期
- 粘贴时必须支持多行文本和 Unicode

建议体验目标：

- 默认行为尽量向 Windows Terminal 靠拢
- 对历史浏览与提示符编辑区要有清晰边界

### 7.8 Unicode、字宽与复杂字符必须稳定

必须正确处理：

- 中英文混排
- 全角字符
- emoji
- surrogate pair
- combining marks
- ZWJ 序列
- East Asian Ambiguous 字符

验收标准：

- 光标列定位与可见字符网格一致
- 清除、插入、删除、覆盖不会破坏复杂字符边界
- 行尾与宽字符邻接场景不产生明显错位

### 7.9 Resize 行为必须像终端，而不是像普通文本框

必须满足：

- 调整窗口尺寸时，当前可见屏幕内容与终端光标位置保持可理解的一致性
- 备用屏 resize 行为与全屏程序预期一致
- 主缓冲区至少要保证“不会因 resize 明显破坏视图语义”

长期必须实现：

- 历史内容 reflow

未实现完整 reflow 前，必须满足：

- 不得出现明显错位、覆盖、虚假跳屏
- 文档中要清楚标记这是临时能力边界

### 7.10 稳定性必须优先于技巧性补丁

必须满足：

- 退出 shell、关闭窗口、销毁控件、停止 PTY 时无死锁
- 大量输出下界面不卡死
- 高频刷新程序下无明显 UI 事件争抢
- 文本量增长下有明确 scrollback 限制策略

## 8. P0 / P1 / P2 优先级

### P0：不解决就不可能“对标体验”

- 缓冲区、视口、光标、选区四者职责解耦
- 完整交互状态机
- 主屏/备用屏识别修正
- 浏览历史与跟随输出行为修正
- 光标独立语义化
- 输入、粘贴、IME 基础稳定

### P1：决定“像不像成熟终端”

- 完整选择/复制/右键策略
- 鼠标协议系统化支持
- Unicode/emoji/复杂字宽稳定
- resize 语义稳固
- scrollback 管理与性能边界

### P2：决定“能否长期维持体验”

- 完整 reflow
- 更强的测试基建
- 更高性能渲染路径
- 平台一致性细节补齐

## 9. 推荐模块边界

无论使用哪种内部实现，最终都应收敛出以下模块边界：

### 9.1 `TerminalBuffer`

职责：

- 保存主缓冲区与备用缓冲区
- 保存字符单元、属性、宽度信息
- 提供行、列、块、scrollback 基础操作

### 9.2 `ViewportModel`

职责：

- 定义当前可见窗口范围
- 维护 `FollowOutput` 与 `BrowseHistory`
- 管理滚动条映射

### 9.3 `CursorModel`

职责：

- 终端光标位置
- 光标可见性
- 光标样式
- IME 锚点信息

### 9.4 `SelectionModel`

职责：

- 选择起止点
- 块选择或线性选择策略
- 复制序列化规则

### 9.5 `InputTranslator`

职责：

- Qt 事件转 VT 输入
- 键盘模式
- 鼠标模式
- bracketed paste
- IME 事件映射

### 9.6 `RenderLayer`

职责：

- 只负责绘制
- 不拥有终端业务状态
- 支持文本层、选择层、光标层分离

## 10. 必须建立的测试矩阵

只有代码测试不够，必须建立“行为测试矩阵”。

### 10.1 Shell 与行编辑

- PowerShell + PSReadLine
- `cmd`
- `bash`
- `python`
- `node`
- `ipython`（如可用）
- `Claude Code`

### 10.2 全屏/TUI

- `vim`
- `less`
- `fzf`
- `lazygit`
- 任意可稳定运行的 curses/TUI 示例程序

### 10.3 关键场景

- 输入一行命令后上下翻历史
- 滚动到历史位置后继续输入
- 复制一段历史文本后粘贴
- 进入全屏程序后滚轮、箭头、点击
- 中文输入法候选框定位
- emoji 与全角字符混排
- resize 前后内容稳定性
- 大量输出后滚动与选择

### 10.4 必须自动化的回归用例

至少自动化以下用例：

- 主缓冲区滚到历史后 `syncCursor` 不得抢回到底部
- 备用屏未开启鼠标协议时，滚轮不得伪造方向键
- 备用屏开启鼠标协议时，滚轮必须转发正确 VT 鼠标事件
- 进入备用屏与退出备用屏时，主缓冲区历史与视口恢复正确
- 光标隐藏/显示不会破坏普通文本颜色

## 11. 实现保证机制

“功能都达到”不能靠乐观推进，只能靠一套强制执行的实现机制。后续开发必须遵守本节，否则即使短期功能列表看似在增长，最终也极可能陷入：

- 新功能不断增加，但交互越来越不稳定
- 修一个场景，坏另一个场景
- 功能名义上“支持”，但实际上不可持续维护
- 表面完成度很高，实际无法对标成熟终端体验

### 11.1 先做最难路径可行性验证，不先铺外围功能

在进入大规模实现前，必须先做 spike，验证以下高风险能力在当前路线下是否真实可做：

- 主缓冲区与备用屏缓冲区分离
- 视口与终端光标分离
- 浏览历史模式与跟随输出模式切换
- IME `preedit` 与候选定位
- Unicode / emoji / 宽字符边界
- resize 后内容与坐标稳定性

约束：

- spike 的目的不是做成最终版本，而是验证关键技术路径是否成立
- 若 spike 已证明当前路线在某个 P0 目标上长期依赖编辑器默认语义才能勉强成立，则必须触发架构止损，不得继续向该路线堆功能

### 11.2 每个功能必须有唯一承载层

后续实现中，每个终端能力必须映射到唯一主责模块，不得出现“这个功能在 3 个地方各补一部分”的模式。

强制映射规则：

- 缓冲区语义由 `TerminalBuffer` 主责
- 视口与滚动模式由 `ViewportModel` 主责
- 光标位置与显示策略由 `CursorModel` 主责
- 选择与复制序列化由 `SelectionModel` 主责
- 键鼠输入与 VT 编码由 `InputTranslator` 主责
- 视觉输出由 `RenderLayer` 主责

禁止：

- 用 UI 控件滚动值直接充当终端视口真相源
- 用编辑器真实插入光标直接充当终端光标真相源
- 把同一功能分散在 `event handler + syncCursor + parser callback` 三处拼接

### 11.3 建立能力矩阵，再开始排期

在任何大规模实现前，必须先维护一份能力矩阵。矩阵中的每一项至少包含：

- 功能名称
- 优先级（P0 / P1 / P2）
- 当前状态：`NotStarted` / `SpikeOnly` / `Partial` / `Verified`
- 主责模块
- 依赖项
- 最大风险点
- 自动化验证方式
- 手工验证方式

必须覆盖的首批能力项：

- FollowOutput / BrowseHistory 状态切换
- AlternateScreen 进入/退出
- Claude Code 兼容性
- PSReadLine 兼容性
- Unicode 宽字符稳定性
- IME preedit
- Bracketed paste
- 选择复制粘贴
- resize 基础稳定性
- scrollback 管理

没有进入能力矩阵的功能，不允许口头认定为“后面顺手做”。

### 11.4 设置架构止损线

必须定义明确的“继续补丁”与“必须重构”分界线。

以下任一条件成立时，必须暂停新增功能，转入重构：

- 某个 P0 功能需要长期依赖 `QPlainTextEdit` 或 `QTextDocument` 的编辑器默认语义才能成立
- 同一个功能修复已连续两次引发其他已验收场景回归
- 为实现一个 P0 能力，需要在 3 个以上无统一模型的事件入口中同步维护状态
- `Claude Code`、PSReadLine、`vim` 三类代表性场景不能同时稳定
- 浏览历史、选择文本、输入命令三者之间仍存在根本冲突

止损原则：

- 一旦触发，不允许继续以“先把别的功能补完再说”为理由绕过
- 必须先解决模型问题，再继续扩展功能面

### 11.5 每个阶段都必须有退出条件

阶段不是按“做了多少文件修改”结束，而是按“某一组能力是否已稳定验收”结束。

每个阶段必须明确：

- 本阶段允许解决的问题范围
- 本阶段不允许扩展的内容
- 本阶段必过的自动化测试
- 本阶段必过的手工场景
- 失败时是否触发重构

如果阶段结束时只能说“看起来差不多”，则视为未完成。

### 11.6 所有功能都必须行为化验收

“写了代码”不等于“功能实现完成”。后续所有功能都必须转化为可验证行为。

例如：

- “支持浏览历史”必须变成“滚到历史区后输入不会跳底，回到底部后恢复跟随输出”
- “支持备用屏”必须变成“`vim` 进入备用屏后滚轮、光标、退出恢复均正确”
- “支持 IME”必须变成“中文输入时有 preedit，候选定位正确，commit 后光标不乱跳”
- “支持 Unicode”必须变成“emoji、全角、组合字符在插入、删除、覆盖时无明显错位”

没有对应验收用例的功能，不得标记为完成。

### 11.7 优先保证代表性程序同时稳定

不能按单个工具逐个“专门适配”，而要用代表性程序交叉验证模型是否正确。

首批代表性程序：

- PowerShell + PSReadLine
- `cmd`
- `bash`
- `Claude Code`
- `vim`
- `less`
- `fzf`
- `lazygit`

实现判断规则：

- 如果一个修复只让某个工具更好，但让另外两个代表性程序变差，则该修复不能算成功
- 只有在同一版本中同时稳定通过多个代表性程序，才能说明模型方向正确

### 11.8 任何阶段都不得跳过回归防线

后续每修复一个体验问题，都至少要做两件事：

- 增加一个能稳定复现该问题的回归测试，或补一个可重复的手工测试脚本
- 重新执行 P0 能力回归集，确认未破坏已有体验

必须坚持：

- 先有失败用例，再改行为
- 先读测试结果，再宣称功能达成

## 12. 分阶段落地计划

### 阶段 1：停止体验继续恶化

目标：

- 先封住最明显的交互回归源头

必须完成：

- 定义并实现 `FollowOutput` / `BrowseHistory` / `AlternateScreen`
- 收紧备用屏启发式逻辑
- 修复光标与视口互相抢占
- 修复 `Claude Code`、PSReadLine 的滚动与输入问题

退出标准：

- 用户在普通 shell 与 `Claude Code` 中不再高频感知“终端宿主在捣乱”

### 阶段 2：建立真正的终端语义层

目标：

- 把终端状态从编辑器语义中剥离

必须完成：

- 引入独立 `TerminalBuffer`
- 引入独立 `ViewportModel`
- 引入独立 `CursorModel`
- 让选择、输入、滚动围绕终端模型工作

退出标准：

- UI 控件不再承担终端主状态

### 阶段 3：补齐成熟终端交互

目标：

- 让全屏程序、复杂字符、IME、选择复制达到稳定可用

必须完成：

- 完整鼠标协议
- 完整 Unicode/字宽修正
- 更成熟的选择/复制粘贴行为
- 备用屏与主屏切换完善

退出标准：

- `vim`、`less`、`fzf`、`lazygit`、`Claude Code` 的主要交互稳定

### 阶段 4：追赶高级体验

目标：

- 解决 resize reflow、性能边界、超长历史与复杂渲染

必须完成：

- reflow
- scrollback trimming
- 性能压测
- 大文本量行为验证

退出标准：

- 用户不会因为极端场景而明显偏向使用外部成熟终端

## 13. 验收标准

只有同时满足以下三类验收，才可宣称“体验已达到可对标 Windows Terminal 的水平”。

### 12.1 自动化验收

- 所有 P0 回归测试通过
- 所有新增核心状态机测试通过

### 12.2 手工行为验收

在 Windows 平台下，使用以下程序逐项测试：

- PowerShell
- `cmd`
- `bash`
- `Claude Code`
- `vim`
- `less`
- `fzf`
- `lazygit`

所有关键场景中不得出现：

- 错误滚动模式
- 视口被意外拉回底部
- 错误进入备用屏
- 光标错位或错显
- 中文/emoji 显著错位
- 复制粘贴明显不符合终端直觉

### 12.3 主观体验验收

至少由熟悉 Windows Terminal 的使用者执行体验对比，结论必须满足：

- “我能在这个控件里做日常命令行工作，而不会因为宿主终端本身频繁出戏”

如果评价仍然是：

- “能用，但是不像真正终端”

则视为未达标。

## 14. 实施纪律

后续实现必须遵守：

- 先补状态机，再补单点修复
- 先写回归测试，再改行为
- 避免再往现有代码里叠加大量无统一语义的布尔条件
- 每次修一个体验问题时，要先确认该问题属于哪个状态边界失真
- 若某项体验目标需要与现有 `QPlainTextEdit` 语义正面冲突，应优先保护终端语义，而不是保护编辑器默认行为

---

## 15. 最终判断原则

未来每次设计决策，都用下面这个问题判断：

**“如果用户正在使用 Windows Terminal，这个场景下他们会期待发生什么？”**

如果当前实现给出的答案不同，那么必须证明：

- 这是明确接受的能力边界

否则就应视为缺陷，而不是“实现差异”。

## 16. 附录：当前项目最明显的体验风险点

根据当前仓库状态，已知高风险区域包括但不限于：

- 视口与光标耦合
- 滚轮与输入历史冲突
- 选择后光标同步策略
- `QTextDocument` 作为主终端状态的长期复杂度问题
- resize 后历史排版缺失
- 主缓冲区清屏/回原点重绘与“跟随到底部”之间的语义边界

当前已知已经收敛但仍需持续关注的区域：

- 备用屏启发式识别
  说明：默认 heuristic alternate screen 已关闭，当前以显式 VT 备用屏切换为准；若后续重新引入 heuristic，必须先补程序级回归测试。
- IME `preedit`
  说明：当前项目已接管 preedit 绘制与候选框锚点，但距离“终端原生级 IME 体验”仍有最后一段模型工作要做。

这些点不是零散 bug，而是“未建立完整终端体验语义”的外在表现。
